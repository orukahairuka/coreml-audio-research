現在のブランチの未コミット変更（や雑なコミット）を整理して、適切な粒度のコミットに分割し、feature/ ブランチを作成して PR を立てる。複数の独立した機能が混在している場合は、機能ごとに別々の feature/ ブランチと PR に分ける。

## 全体の流れ

1. 変更の把握
2. PR 単位の判定
3. 各 PR 内のコミットグルーピング
4. feature/ ブランチの作成とコミット
5. PR の作成

## 手順

### 1. 変更の把握

現在のブランチ上の変更の全体像を把握する。

```bash
git branch --show-current
git log main..HEAD --oneline
git diff main --stat
git diff main
git status
git diff
git diff --cached
```

変更が未コミットの場合もコミット済みの場合も、最終的に main との差分全体を整理対象とする。ただし既にマージ済みの PR に含まれるコミットは除外する。

### 2. PR 単位の判定

差分全体を読み、独立した機能や目的ごとに PR を分けるべきかを判断する。

判断の指針:
- **独立した機能は別 PR にする**: 例えば「音声フィルタの追加」と「CI の改善」が混在していたら2つの PR に分ける
- **依存関係があるものは同じ PR にする**: 機能 B が機能 A に依存しているなら、同じ PR にまとめるか、A を先にマージしてから B の PR を作る
- **迷ったら1つの PR でよい**: 無理に分けるより、レビューしやすさを優先する

PR の分割案をユーザーに提示して確認を取る。1つの PR で十分な場合は提示を省略し、コミットグルーピングに進む。

### 3. 各 PR 内のコミットグルーピング

各 PR に含める変更を、コミット単位に分ける。

**粒度の原則: 1コミット = 1つの論理的変更。迷ったら小さく分ける。**

グルーピングの指針:
- **1ファイル1コミットが基本**。同じ論理的変更を複数ファイルにまたがって行う場合（例: enum 定義とそれを使う側を同時に追加しないとビルドが通らない）のみ複数ファイルを1コミットにまとめる
- **分けるとビルドが通らない変更**は1つにまとめる
- **独立した変更は必ず分ける**: リファクタリング、テスト追加、設定変更、新規ファイル追加など
- **同じファイルでも独立した変更は分ける**: 例えばデフォルト引数の削除とエラーメッセージの改善は別コミット
- **関連のない変更を1つにまとめない**

具体例:
- ❌ 「変換スクリプトの出力名変更」で convert_transformer.py + convert_hifigan.py + synthesis_coreml.py を1コミット
- ✅ convert_transformer.py + convert_hifigan.py（同じ変更）で1コミット、synthesis_coreml.py（別の変更）で1コミット

グルーピングの結果をユーザーに提示して確認を取ってから実行する。

### 4. feature/ ブランチの作成とコミット

```bash
# 未コミットの変更がある場合は退避
git stash --include-untracked

# main から新しいブランチを作成（リモートの最新を取得してから）
git checkout main
git pull origin main --ff-only
git checkout -b feature/<適切な名前>

# stash を戻す
git stash pop
```

コミットメッセージとブランチ名は `.claude/rules/git-workflow.md` のルールに従う。

### 5. PR の作成

```bash
git push -u origin feature/<ブランチ名>
gh pr create --base main --title "PRタイトル" --body "PR本文"
```

PR のタイトルと本文:
- タイトルは短く、変更内容を端的に表す（日本語 OK）
- 本文には変更の概要と、必要に応じて背景や注意点を書く
- 過度にフォーマットに拘らず、レビューアーが理解しやすいことを優先する
- 複数 PR がある場合、関連する PR のリンクを本文に記載する

### 完了後

すべての PR の URL をユーザーに伝える。

## 注意事項

- 元のブランチの変更は壊さない。新しい feature/ ブランチに変更をコピーする形で整理する
- コミットメッセージ・ブランチ名のルールは `.claude/rules/git-workflow.md` を参照する
- ユーザーに PR 分割とコミットグルーピングの確認を取ってから実行する
- コミットの粒度に迷ったら「このコミットだけを revert しても意味がある単位か？」を基準にする
- PR の分割に迷ったら「この PR だけをマージしても意味がある単位か？」を基準にする
