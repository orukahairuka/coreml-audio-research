# モデルロード時間 実機計測（初回 / キャッシュ後）

本番12セル（精度3 × 計算デバイス4, shape=fixed262）で、**セルをタップしてから音を出す準備＝
Encoder + Decoder + HiFi-GAN の3モデルロードに何秒待たされるか**を実機計測した。
「その場でインタラクティブ」の成立を問う [feasibility study](../2026-06-08/interactive-feasibility-direction.md)
の「速い」軸を埋めるのが目的。

- **端末**: iPhone (3) / iOS 26.6
- **日付**: 2026-06-10
- **回数**: n=2（再現確認）。各回ともアプリ再インストールでコンパイルキャッシュをまっさらにし、合成を一度も実行する前にスイープ
- **ハーネス**: `ModelLoadTimingTester`（`feature/model-load-timing`）。1セルにつき各モデルを「初回 → キャッシュ後」の2回ロードして計時
- **生データ**: `metrics/load_timing_iPhone_3_20260610.csv`（run1）, `metrics/load_timing_iPhone_3_20260610_run2.csv`（run2）※ metrics/ は gitignore

## なぜ「初回」と「キャッシュ後」を分けるか

`.mlmodelc` はビルド時にコンパイル済みだが、`MLModel(contentsOf:configuration:)` は各 computeUnits 向けに
**初回だけ特殊化（プラン生成）**を走らせる。これが初回ロードの主成分。2回目以降は OS のコンパイル済み
キャッシュに当たるため速い。ユーザー体感では「各セルを初めて触った瞬間だけ遅く、戻ってきたら速い」。
セルの切り替えはユーザー操作なので、こちらが問うのは「選んだ瞬間の待ち時間」＝ロード時間とした。

## 読み取れること

### 1. キャッシュ後は全セル ~0.1秒に収束 — 差を生むのは100%「初回コンパイル税」

`total_cached_ms` はデバイス・精度によらず 77〜210ms。つまり同じセルを2回目以降ロードする
（行き来する）体験ではどのセルも実質即時。**体感差を生むのは各セルを初めて触った瞬間の特殊化のみ。**

### 2. 犯人は HiFi-GAN の ANE 特殊化

遅いセルの初回内訳を見ると、Encoder/Decoder は1秒台なのに **HiFi-GAN 単体で 32〜67秒**を占める。

| セル | HiFi-GAN 初回 (run1 / run2) |
|------|-----------------------------|
| Float16 cpuAndNE | 54.6s / 58.0s |
| Float16 all | – / 34.4s |
| Int8 cpuAndNE | 54.8s / 66.6s |
| Int8 all | 32.2s / 35.5s |

ANE 向けのプラン生成だけが異常に重い。HiFi-GAN（多層の転置畳み込み）の ANE コンパイルがボトルネック。

### 3. Float32 の ANE セルが速いのは「ANE を使っていないから」

Float32 × {cpuAndNE, all} は1秒前後で済む。これは [Float32 が ANE 非対応](../2026-05-19/stability-matrix-analysis.md)
で、ANE を要求してもフォールバックして実際には ANE を使わないため。"速い" の中身が他の精度と別物。

### 4. クラッシュは非決定的、遅さは確定的

- **遅さは完全再現**: F16/Int8 × {cpuAndNE, all} は2回とも 34〜69秒。ANE 特殊化が重いのは確定的な性質。
- **クラッシュは揺れる**: Float16 × all は run1 でロード中にプロセスごと OS に kill された（画面が真っ暗に）が、
  run2 では 36.7秒で完走。Float16 × cpuAndNE も最初の試行でハング、その後は完走。
- つまり ANE セルは「**毎回必ず落ちる**」ではなく「**数十秒かかり、かつ時々プロセスごと落ちる**」。
  どちらに転んでも "速い" は不成立。

> 補足: この不安定性のため、ハーネスにはクラッシュ再開（CSV 固定名 + 記録済みセルのスキップ + クラッシュ痕跡を
> `crashed_on_load` で記録）と90秒ウォッチドッグ（`load_timeout`）、安全側(cpuOnly→cpuAndGPU)を先・
> ANE側(cpuAndNE→all)を後に回す順序を入れた。落ちても再起動でボタンを押し直すだけで続きから埋まる。

## 成立マップへの含意

「速い」軸で**不成立になった4セル（Float16/Int8 × {cpuAndNE, all}）は、
[clipping で「壊れない」軸も不成立だったセル](../2026-05-19/stability-matrix-analysis.md)と完全一致**する。
[dispatch マップ](../2026-05-19/compute-plan-analysis.md)で HiFi-GAN が NE dispatch されるのも同じ4セル。

→ **「壊れない」軸と「速い」軸が、同じ ANE セルを同時に棄却している。** ANE 経路に HiFi-GAN を載せると
振幅が壊れ、かつロードが数十秒かかる/落ちる。逆に **CPU/GPU 系8セルは全精度で初回1秒前後**＝
インタラクティブ成立の候補は CPU/GPU 側に絞られる。

## 残課題

- 数十秒の ANE コンパイルがアプリ初回起動時に一度だけなら許容か、UX として致命的かの判断（プリコンパイル/事前ウォームの可否）
- クラッシュの再現条件の切り分け（メモリ圧 / jetsam / E5RT のどれか）— ただし "速い" 不成立の結論は変わらない
- 「使える音」軸（聴感）はまだ別途
